# Open Model(로컬 LLM) 활용 — 라이선스 및 벤치마크 노트

이 문서는 `local_llm`(Ollama 기반) 백엔드가 사용하는 오픈 모델의 역할, 선택 이유, 라이선스
조건, 실사용 비교를 정리한다. 대상 모델은 `config.yaml`의 `local_llm.model`에 설정된 값을
기준으로 하며, 배포 환경에 따라 실제 모델명은 다를 수 있다(예: `qwen2.5-coder:7b`,
`qwen3.5:9b-mlx` 등 Qwen 계열).

## 1. 이 프레임워크에서 로컬 오픈모델의 역할

`agent.py`의 라우팅 설계상 로컬 모델은 **모든 작업을 처리하는 범용 코딩 모델이 아니라, 아래
두 가지로 역할이 한정된 1차 처리 계층**이다.

1. **잡담/설명 판별** (`router.is_chatter`) — 입력이 코딩 작업인지 잡담인지 가볍게 분류.
2. **설명 모드 응답** (`agent.py`의 `explain_task`, `_is_general_information_request`) — 코드를
   수정하지 않고 질문에 답하는 읽기 전용 요청.

실제 **코드 수정(파일 작성/실행 등)을 동반하는 작업은 `openrouter`(또는 그마저 실패 시 외부
CLI)가 기본 경로**이며, 로컬 모델은 이 앞단에서 가벼운 요청을 걸러내 외부 API 호출 자체를
줄이는 필터 역할을 한다. 이는 대규모 언어모델 API 비용/지연을 아끼는 동시에, 작업 성격을 아직
모르는 초기 판별 단계에서 사내 코드를 외부로 보내지 않기 위한 설계다.

## 2. 모델 선택 근거

| 후보 | 비고 |
|---|---|
| **Qwen2.5-Coder (7B, Apache-2.0)** | 코딩 특화 파인튜닝, 로컬(Ollama)에서 7B급으로 실시간 응답 가능한 크기, 상업적 사용 제약이 없는 라이선스 |
| Llama 3.1 8B (Llama 3 Community License) | 코딩 특화도가 상대적으로 낮고, 재배포 조건(월간 활성 사용자 수 등)이 있는 라이선스라 배제 |
| DeepSeek-Coder | 성능은 유사하나 팀 내 검증 사례 부족으로 2순위 후보로 보류 |

Qwen2.5-Coder를 선택한 이유:
- **라이선스**: Apache License 2.0 — 상업적 사용, 수정, 재배포에 제약이 없고 별도 고지 의무도
  가볍다(NOTICE 파일 보존 정도). 사내 프로젝트(gen3 등)에 투입하는 데 법적 리스크가 낮다.
- **경량 판별용 모델**(`local_llm.router_model`, 예: `qwen2.5:3b`)도 동일 계열로 통일해 배포·
  운영 복잡도를 낮췄다.
- Ollama로 로컬 배포가 쉽고, 7B Q4 양자화 기준 RAM 16GB급 워크스테이션에서 실시간 판별에
  충분한 속도가 나온다.

## 3. 라이선스 검토 요약

- **Qwen2.5 계열 라이선스**: Apache License 2.0 (Qwen 팀이 2.5 세대부터 이전 세대의 Tongyi
  Qianwen 전용 라이선스에서 Apache-2.0으로 전환).
- 확인 사항:
  - [x] 상업적 사용 허용
  - [x] 사내 프로젝트(gen3 등)에서 소스 비공개로 활용 가능
  - [x] 재배포 시 라이선스 사본·저작권 고지 포함 조건만 있음 (Copyleft 아님 — 파생물 소스공개
        의무 없음)
  - [ ] 모델 가중치 자체의 재배포는 하지 않음(로컬 Ollama 캐시로만 사용, 별도 배포 채널 없음)
        — 재배포 조건은 현재 사용 범위에서 해당 없음
- **결론**: 사내 폐쇄망/사내 프로젝트 코드에 대해 로컬 추론 목적으로 사용하는 현재 범위에서는
  라이선스상 제약이 없다. 향후 모델 가중치 자체를 외부에 재배포하거나 파인튜닝 결과물을
  배포할 계획이 생기면 그 시점에 라이선스 재검토가 필요하다.

## 4. 보안 검토 요약

- 로컬 모델은 사내망 안(`local_llm.base_url`)에서만 통신하며, Ollama 서버가 외부 인터넷에
  노출되지 않는 것을 전제로 한다(배포 시 방화벽/사설 IP 대역 확인 필요).
- 잡담 판별·설명 응답 단계에서 다루는 데이터는 `harness/metrics.py`의 `redact_sensitive()`로
  password/token/api_key/license_key 패턴이 로그 기록 전 마스킹된다(자세한 마스킹 범위는
  README 8절 참고).
- 로컬 모델 자체가 별도 학습 없이(파인튜닝 없이) 순수 추론에만 쓰이므로, 사내 코드가 모델
  가중치에 흡수되어 유출될 가능성은 없다.

## 5. 간이 벤치마크 (정성 비교)

정식 정량 벤치마크(HumanEval 등 공개 데이터셋 기반)는 아직 수행하지 않았다. 대신 아래는 실제
gen3 프로젝트 실사용 중 로컬 모델과 OpenRouter(Claude 계열)의 처리 결과를 비교한 정성 평가다.

| 작업 유형 | 로컬(Qwen 계열) | OpenRouter(Claude) |
|---|---|---|
| 잡담/인사 판별 | 정확도 높음, 응답 3초 이내 | (해당 경로 미사용) |
| 프로젝트 1문장 요약 설명 | 준수한 요약, 20~30초 | 더 정교하지만 과할당 |
| 다중 파일 리팩토링/스키마 설계 | 시도 시 반복 루프·품질 저하 관찰 | 안정적으로 완료 |
| SECS/GEM 프로토콜 등 도메인 특화 로직 | 도메인 지식 부족으로 부적합 | 컨텍스트 이해 우수 |

**결론**: 판별·경량 설명 작업에서는 로컬 모델의 응답 품질과 속도가 실용적 수준이나, 여러 파일에
걸친 구조 변경이나 도메인 특화 로직(SECS/GEM 등)에는 로컬 7B급 모델이 아직 충분하지 않다.
이 격차가 현재 라우팅 설계(로컬=판별/설명, OpenRouter=코드 작업)의 실증적 근거다.
